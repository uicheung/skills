# Generate Tech Sharing

本技能旨在帮助团队成员快速产出**高质量的技术分享文档**，或对已有草稿进行深度润色，以便于在团队内部进行知识沉淀。

核心原则：**结构清晰、内容可信、阅读顺畅、可长期沉淀**。

典型应用场景包括：进行技术分享、构建团队知识库、沉淀最佳实践文档。

## 流程触发场景

### 触发条件
1. 用户表达出需要进行技术分享的意图
2. 发现用户解决了某个复杂问题时并被你捕捉到这一动作时（需询问用户是否进行技术分享，当用户同意时方可触发）

### 初始引导话术
为了帮用户撰写技术分享，需要执行以下三个阶段的任务

## 第一阶段：背景信息收集
*目标*:对齐 Agent 和 用户之间的信息差，为后续精准生成大纲和内容做准备

### 初始提问
1. 本次的文章主题是什么？
2. 希望用户阅读后产生何种效果？
3. 是否有特定的代码片段其它内容必须向读者展示？
4. 是否有特定的模板或特定的格式要求？
5. 还有其他约束或关键的背景信息或参考资料需要补充吗？

如果用户提供资料不能直接读取时，你可以判断是否必要或尝试使用其他 MCP 工具、Agent Skills 等方式来加载内容。


### 阶段结束条件
当Agent 能基于已收集到的信息，提出边缘场景和方案取舍的问题，且无需用户解释时， 或用户主动要求结束该阶段时，即判定改阶段结束。

### 阶段过渡话术
询问用户是否还需要补充更多信息，或是否可以进入文章大纲草拟阶段。若用户需要补充，则继续收集，否则进入下一阶段。

## 第二阶段：文章大纲草拟
*目标*：通过头脑风暴对已有信息总结，列出文章大纲。

1. 根据文章主题，提出循序渐进的大纲规划
2. 草拟每段的内容简要概括
3. 对于用户要求必须展示的信息所处区域的规划
4. 用户的需求可能很片面，需要结合 Agent 的知识充实大纲，但不可太偏移主题
5. 至少需头脑风暴出 3 - 5 个大纲方向，每份大纲列出编号可供用户选择
6. 大纲及后续内容编写需要写入本地文件

### 阶段结束条件
当 Agent 向用户列出你的大纲并向经用户选择后，判定此阶段结束，否则询问用户优化方向。


## 第三阶段：文章内容编写
*目标*: 分段编写文章内容，并迭代优化内容

### 步骤一：收集用户更期望通过何种文笔风格来编写文章
若用户无特殊要求，你需要使用优美的文笔来撰写文章，不能写出类似文档的冰冷文章，但也不宜过于浮夸和不切实际。

### 步骤二：遵循文章编写规范
可以询问用户是否有其他规范要求并简短说明已经内置了一些规范要求，并遵守下列规范及用户附加规范。

1. 固定的技术分享文章头格式
```markdown
| 课题 | <文档标题> |
|------|------------------------------|
| 作者 | - |
| 审定 | - |
| 目的 | <文档目标简述> |
```

2. 引用内容需在文章末尾注明
```markdown
## 参考资料
- https://vuejs.org
- https://www.typescriptlang.org
```

3. 文章中不能包含莫名 emoji，和一些颜文字（✓ x 等）
4. 不应存在不实内容
5. 章节之间行文逻辑自洽，不能存在矛盾和章节间不连贯
6. 输出内容需要是中文
7. 如有章节编号信息不应在文章中体现，仅可用于内容编排
8. 文章内容中允许使用 mermaid、或 Agent 能使用的其他草图等生成工具
9. 当有代码片段时，需要仅提取核心内容
10. 文章总体字数需在 5000–10000 汉字左右，适配 10-20分钟的阅读时间
11. 技术分享不能泄漏业务系统敏感信息
12. 章节标题不宜过长，应该是一个章节内容的概括或题引，标题中也不建议使用如开篇、结尾、总结等内容
13. 文章开头可以通过讲述一个小故事或一些诙谐的开场

### 步骤三：输出当前章节内容，并在用户的协助下迭代优化内容
1. 使用其他工具（如命令行工具等）来核实总体字数是否达标
1. 文章完成编写时，需要询问用户意见，并进行调整直至满意
2. 在新增章节或修改内容时，*禁止重新粘贴整篇文章，仅允许通过文本替换功能来完成修改*

### 阶段结束条件
当用户对所有章节内容都满意后，则判定阶段结束。并向用户简段告知文章存储路径。
